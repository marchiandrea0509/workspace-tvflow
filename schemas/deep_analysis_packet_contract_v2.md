# Deep Analysis Packet Contract v2

Default workflow for read-only Bitget deep analysis.

## Key decisions

- Primary task: analyze screenshots / visible chart structure first; use raw OHLCV/export data only as fallback when screenshot levels/data are unclear, and to validate current price, ATR, sizing, margin/leverage/liquidation, liquidity, and execution feasibility.
- Data priority: (1) screenshots / visible chart structure, (2) user notes / visible levels as secondary context, (3) Bitget OHLCV/export/ticker/execution fallback and validation, (4) TradingView OHLCV/export cross-check only if needed.
- Candidate scans, static optimisation scan output, OHLCV-derived value zones, and old packet ladder levels are fallback / secondary validation only. They must not be used as the main source of trade levels and must never override visible 1D / 4H / 1H chart structure.
- TradingView evidence: generated by default for deep analysis; prefer existing Playwright/browser capture or TradingView Desktop CDP when available.
- Focus window: 4H last 40 candles, 1H last 80 candles, 1D last 80 candles; older data only for major HTF levels.
- Screener: summary/context only; current context `OC Hybrid Edge Screener v11.6.x`.
- No hard screener-score threshold.
- Target planned risk: 100 USDT by default.
- Cap: 1500 USDT **max margin** is a leverage-sizing target, not a total-notional cap and not a chart-quality reject by itself. Packet/report logic must recalculate required leverage up to 20x before treating margin as a blocker. Reject for margin only when the ticket cannot fit under the margin cap at <=20x or liquidation safety fails.
- Existing live orders/ladders must be evaluated separately from fresh placement. If the existing ladder is structurally valid and risk-capped, do not mark it invalid only because its current exchange leverage produces >1500 USDT all-filled margin; report the leverage that would fit and state that any leverage change requires separate explicit user confirmation.
- Live execution: excluded; requires separate explicit user request.
- Rejection audit: every rejected side/option/candidate and every broken rule must be justified with the exact rule/gate, observed value, required value/threshold, why it blocks orderability, and what would fix it.
- Trade-family framework: final reports use Andrea's A/B/C/D master prompt. A = BEST QUALITY PULLBACK, B = BEST FILL-PROBABILITY PULLBACK, C = BREAKOUT/BREAKDOWN, D = OC EXECUTION WRAPPER (`VIRTUAL_OCO` or `COMBO_100`) only if useful and valid. A/B/C are alternatives unless D explicitly wraps them.
- Shallow fill-probability audit: every rejected shallow fill-probability leg must show candidate entry, SL, nearest TP, next TP, R:R to nearest TP, R:R to next TP, and accepted/rejected reason.
- Universal resting-entry distance gate: every resting `DIP_LADDER`, `SELL_RALLY`, or resting `AUTO / SINGLE_LIMIT_PULLBACK` candidate must pass the no-chase current-price distance filter before combo construction and A/B pullback ranking. Long buy limits need distance `(current - entry) / ATR4H >= 0.25`; short sell limits need `(entry - current) / ATR4H >= 0.25`. In strict contexts (1H against/corrective, near major S/R, market chase disallowed, or fill-probability use), require `>= 0.50`. Failing entries must be stored in `near_market_rejected_entries` and excluded from A/B pullback tickets.
- If requested side is `AUTO`, the packet builder must infer a working LONG/SHORT side from screener context or 1D/4H trend for directional fib/value-zone mapping, and expose `requested_side` plus `analysis_side_inference`. Final judgment remains screenshot-first.
- Compact packets must expose `detected_level_map` and reports must include a `Detected Level Map` section before narrowing levels into tickets. Reports must not hide intermediate detected levels merely because only some become actionable order levels.
- Packets must expose `impulse_zone_comparison` for both LONG and SHORT flows: selected/confirmed 4H impulse fibs plus active/fresh-high or fresh-low visual impulse fibs when available, so final reports can compare swing choices. The comparison must not be limited to the nearest compact levels; it must surface recent 4H parent-swing anchors and broader/stale HTF alternatives that are farther from current price. Final reports must print `PB impulse used` with selected swing, reason, and approximate 23.6/50/61.8 levels when useful, then compare any materially different local/recent-parent/stale-broad alternative. This is mandatory because wrong impulse selection is a known recurring regression.
- Near major 4H/1D resistance/support, final reports must not treat the packet `selected_builder_impulse` or static optimisation result as authoritative if screenshots expose a better visible 4H parent swing. Prefer the recent visible 4H parent swing that caused the current shelf/breakdown/breakout, commonly ~3-7 ATR4H, over both the latest tiny impulse and stale over-broad 1D pivots unless screenshots justify otherwise. Static scan failure is a warning/audit input, not a veto over screenshot-valid A/B levels.
- Detected level maps should be dense enough to audit against screenshot/GPT-style analysis: include nearest-to-farthest 1H/4H/1D supports/resistances, intermediate shelves, fresh highs/lows, and major HTF levels before narrowing to tickets.

Final V2 reports must briefly reject any invalid A/B/C/D family and include a rejection-audit table whenever the final decision is WAIT/NO_TRADE or a material candidate is rejected. Use `static_ticket_reject_reasons`, `warnings`, `static_optimisation_scan.best_candidate.reject_reasons`, `rejected_candidate_examples_compact`, and screenshot judgment as source evidence.

Final V2 reports must follow the active master prompt's family structure:

- A) `BEST QUALITY PULLBACK` — only provide if valid; otherwise briefly reject.
- B) `BEST FILL-PROBABILITY PULLBACK` — only provide if valid; otherwise briefly reject.
- C) `BREAKOUT / BREAKDOWN` — only provide if the breakout/breakdown gate passes; otherwise briefly reject.
- D) `OC EXECUTION WRAPPER` — include `VIRTUAL_OCO` or `COMBO_100` only if useful and valid.

If a family is rejected, include the exact failing reason and key failed metric where available: R:R, SL validity, TP realism/open space, fakeout/chase risk, margin, leverage, liquidation safety, or lack of useful OC benefit.

Full output format is mandatory for every deep analysis. Reports and chat replies must include the active master prompt's compact 5-day swing-plan structure: header/classification, Context and State, Key Levels, Pullback Impulse Used, A/B/C/D options, Orderability traffic-light table, Risk Sizing/Tickets, and Final Verdict. Do not collapse the answer into only chart context plus final verdict or only actionable ticket levels. Orderability must include an explicit traffic-light table with observed values and limits whenever liquidity/exchange gate data is available; do not summarize it as only OK/RED. A saved report is not complete if the chat reply omitted the impulse map, valid ticket tables, or gate table.

Every `VALID` A/B/C option must include a complete Trade Plan Ticket, including `PLACEABLE_CONDITIONAL_ONLY` options. Conditional means the setup/ticket is valid but waits for the stated trigger/rejection before placement; it does not allow omitting the ticket. Required ticket columns exactly: `Leg | Entry | Type | Qty | Notional | SL | Loss | TP | Profit | RR | Trigger`. Each row must include its own entry, quantity, SL, TP, estimated loss, estimated profit, R:R, and trigger condition. If exact sizing cannot be calculated, provide an approximate ticket using current symbol price and entry/SL distance and mark values approximate. `REJECTED` and `NOT_PLACEABLE_YET` options do not get tickets, but must include rejection/fix details.

Evaluate A/B pullback families first, then independently test C breakout/breakdown using the breakout gate. A quality logic tries 1-2 cleanest PB levels first and uses 3 legs only if all 3 are high-quality/not forced; if one strong level survives, classify `SINGLE_LIMIT_PULLBACK`. B fill logic maximizes fill mainly by adding an earlier valid shallow L1 above/near the A-quality zone. Attempt up to 3 legs, but prefer extra shallow/mid levels over deep L3 if deep L3 forces materially wider SL or changes core invalidation. A deep L3 is allowed only if still within the same thesis and without moving SL beyond clean structural invalidation unless deeper invalidation is clearly visible/coherent; reject L3 if it only exists to keep 3 legs. For shorts, reverse with `SELL_RALLY` using LH->LL impulse/retrace into resistance. Only add D when already-valid families benefit from OC control. Prefer `VIRTUAL_OCO` when pullback and breakout are alternative entries into the same thesis; use `COMBO_100` only when both may fill, share coherent invalidation, all-filled R:R remains good, combined risk is <=100 USDT, and margin can fit at <=20x with liquidation safely beyond SL.

Never mark a saved report complete if the chat summary omitted mandatory ticket tables for valid options. Prefer printing valid tickets in chat. If a chat surface forces a shorter summary, the chat must say: `Ticket table omitted from chat, full ticket available in saved report.`

## Static 4H ladder rule

Deep Analysis v2 may build static OC 4H pullback ticket candidates for validation: `DIP_LADDER long`, `SELL_RALLY short`, or `AUTO / SINGLE_LIMIT_PULLBACK` when exactly one valid pullback level survives all static checks. These candidates are secondary validation only; the final trade levels must be designed from the 1D / 4H / 1H screenshot structure. If generated value zones or static scan candidates conflict with visible structure, screenshots win. Do not choose old pivots merely because they are confirmed. In fresh breakouts the impulse may be `ACTIVE_BREAKOUT_UNCONFIRMED_HIGH/LOW` using the live/latest 4H/1H extreme and the latest meaningful pullback base instead of waiting for a confirmed pivot extreme. Breakout-retest structural levels are direct entry candidates. Tickets use 1-3 legs maximum, a common structural SL with 0.25-0.50 ATR buffer, fixed TP/SL at order creation, and risk splits by risk not quantity. Each ladder leg must have its own meaningful TP; do not assign the same TP to all legs. R:R is calculated per leg with that leg's TP, and all-filled R:R is summed reward divided by summed planned risk. A valid ticket must remain risk-controlled near the target risk if all entries fill and price immediately goes to SL. No ticket may depend on future cancellation, trailing, SL movement, or post-fill adjustment. Hot RSI / near major S/R blocks market chasing and activates the stricter 0.50 ATR4H resting-entry distance gate; it is not a hard reject for deeper valid resting pullback limits when static checks pass.

Structural SL hierarchy rule: candidate reports and packet validation must justify the exact 4H parent swing and invalidation level used for SL. A tighter SL is invalid when it leaves nearby same-side 1H/4H/1D structure beyond the stop that the thesis could still validly reject from (shorts: overhead resistance / lower-high shelves; longs: lower support / higher-low shelves). However, do not force SL beyond every older/stale level when the current lower-high/higher-low already invalidates the immediate 4H thesis; screenshot structure controls whether reclaiming that nearer level breaks the setup. If a wider structural SL is truly required, recompute R:R; if a shallow fill-probability leg fails, remove it, convert to a deeper 2-leg plan, SINGLE_LIMIT_PULLBACK, or WAIT. Far TP targets beyond nearer major levels require the nearer level to be assigned to an earlier leg or a confirmed break/retest path; do not use far targets to make weak shallow legs pass.

Final Discord delivery must send the analysis text first, then attach only the full-resolution 4H and 1D screenshots as separate follow-up messages, one image per message, 4H first and 1D second. 1H remains mandatory analysis evidence but should not be attached unless explicitly requested. If any analysis screenshot is missing, unreadable, or not used, say so explicitly.
